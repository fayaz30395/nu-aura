| P0-001 | `/recruitment` | EMPLOYEE | Employee could access full NU-Hire recruitment page with admin data.    | EMPLOYEE has `RECRUITMENT:VIEW` but should not access admin hub. AuthGuard's route-level `anyRole` check + page-level role guard. | **FIXED + VERIFIED** — Route config `anyRole` + page-level `hasAnyRole` guard after all hooks. Browser-verified: EMPLOYEE sees "Access Restricted" screen. |
| P0-002 | `/performance` | EMPLOYEE | Employee could access full Performance Management hub with admin stats. | EMPLOYEE has `REVIEW:VIEW` but should not access admin hub. Same dual-guard approach.                                             | **FIXED + VERIFIED** — Route config `anyRole` + page-level guard. Browser-verified: EMPLOYEE sees "Access Denied" from AuthGuard.                          |

**Recommended fix for P0-001 and P0-002:**

1. **Option A** (preferred): Create separate admin-level permissions (`RECRUITMENT:VIEW_ALL`,
   `REVIEW:VIEW_ALL`) for the admin hub routes. Keep `RECRUITMENT:VIEW` and `REVIEW:VIEW` for
   self-service.
2. **Option B**: Change route guards to require higher permissions (`RECRUITMENT:MANAGE`,
   `REVIEW:APPROVE`) and add a separate self-service route.
3. **Option C**: Add `anyRole` restriction to the route configs requiring at least `HR_ADMIN`/
   `HR_MANAGER`/`RECRUITMENT_ADMIN` for `/recruitment` and at least `HR_ADMIN`/`HR_MANAGER`/
   `MANAGER`/`TEAM_LEAD` for `/performance`.

**Files to modify:**

- `frontend/lib/config/routes.ts` lines 224-236 (route permission requirements)
- Backend Flyway migration for EMPLOYEE role permissions
- `frontend/lib/hooks/usePermissions.ts` add new permission constants if using Option A
