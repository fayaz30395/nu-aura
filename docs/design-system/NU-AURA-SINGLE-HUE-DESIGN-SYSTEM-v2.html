01 — Philosophy

Design Philosophy

Color is a functional signal, not decoration. 80–90% of the UI uses the tonal blue scale. Semantic colors (green, red, amber, cyan) are reserved exclusively for meaning. The system balances a controlled tonal palette with strict semantic separation.

Core Principles

1. Color = Meaning — Every color choice communicates function, not aesthetics.

2. Consistency over Creativity — Uniform application across 237 pages and 4 sub-apps.

3. Reduce Noise, Not Signal — Monochromatic surfaces let data and status colors pop.

4. Design for Scanning — HR managers scan payroll tables, attendance grids, and approval queues at speed.

What Changes from Current System

Changed Primary accent shifts from #0057FF to #4d77ff (softer, more tonal)

Changed Neutral gray scale gains a subtle blue tint (desaturated blue-gray)

Kept Semantic colors (success, error, warning, info) remain independent

Kept CSS custom property architecture, dark mode via .dark class

Kept All existing component APIs (Button, Card, Table classes)

New Blue-tinted neutral scale for cohesive monochromatic feel

Critical Rule

Blue-500 (#4d77ff) fails AA for small text on white. Use blue-600 (#3a5fd9) for text links and inline text. Blue-500 is safe for buttons (white text on blue bg, large text) and icons (3:1 threshold). Neutral-400 is only valid for captions/micro-labels (12px+, large text threshold).

Inversion Rules

Light bg (n-50) → Dark bg (n-950). Light card (white) → Dark card (n-800). Light text (n-900) → Dark text (n-100). Accent blue-500 → blue-400 (brighter). Semantic status colors use rgba backgrounds with lighter text variants.

09 — Mantine Integration

Theme Override Strategy

Mantine's theme system consumes CSS custom properties. The single hue change flows through without touching component code.

// lib/theme/mantine-theme.ts — Color tuple update accent: [ '#f0f4ff', // 0 — blue-50 '#dce4ff', // 1 — blue-100 '#b8c9ff', // 2 — blue-200 '#94adff', // 3 — blue-300 '#7092ff', // 4 — blue-400 '#4d77ff', // 5 — blue-500 (primary shade) '#3a5fd9', // 6 — blue-600 '#2a48b3', // 7 — blue-700 '#1c328d', // 8 — blue-800 '#101e66', // 9 — blue-900 ], primaryShade: { light: 6, dark: 4 }, // light=6 (blue-600) for AA compliance on white // dark=4 (blue-400) for visibility on dark surfaces
Key: primaryShade: 6 (not 5)

Mantine uses primaryShade for all primary-colored components (Button, Checkbox, Switch, Tabs active indicator). Setting it to 6 (blue-600, #3a5fd9) ensures all Mantine components automatically pass WCAG AA for white text, while blue-500 remains available for large elements and backgrounds.

10 — Governance

Enforcement Rules

✕

No raw hex in code. Every color reference must go through a CSS variable or Tailwind token. Enforced via ESLint no-restricted-syntax rule matching hex patterns in className strings.

✕

No random colors. Only the blue tonal scale + semantic colors are permitted. Legacy aliases (teal, jade, purple, violet, cyan in Tailwind) must be removed or remapped to the blue scale.

✓

All colors via tokens. Use var(--accent-primary), var(--text-secondary), bg-[var(--bg-card)] — never bg-blue-500 or text-gray-700 directly.

✓

Semantic colors are immutable. Success is always green, error is always red, warning is always amber. These never adapt to the primary hue.

✓

Consistent elevation mapping. Level 0 = border only (inputs, static cards). Level 1 = subtle shadow (interactive cards). Level 2 = medium shadow (hover states, dropdowns). Level 3 = strong shadow (modals, drawers).

✕

Never use blue-500 for small body text on white. It fails WCAG AA (3.7:1). Use blue-600+ for text, blue-500 for buttons/icons only.

11 — Rollout Strategy

3-Phase Migration

Zero-downtime migration leveraging the existing CSS variable architecture. No component API changes required.

  1. Phase 1: Token Foundation (1 day) Update globals.css (CSS variables), tailwind.config.js (color scales), mantine-theme.ts (color tuples + primaryShade), and design-system.ts (token exports). Run npx tsc --noEmit. This single change propagates to all 237 pages through the variable cascade.
  2. Phase 2: Audit & Fix Hardcoded Values (2–3 days) Grep for hardcoded hex values, legacy Tailwind color classes (bg-indigo-*, text-slate-*, bg-sky-*), and inline styles with color. Replace with CSS variable references. Update compact-theme.ts gradient definitions. Update apps.ts sub-app gradients.
  3. Phase 3: Visual QA & Polish (2 days) Browser test all 237 pages in both light and dark mode. Verify contrast ratios on real screens. Check chart colors, status badges, sidebar, header, modals, dropdowns. Fix edge cases. Update screenshots in docs.
12 — Data Visualization

Chart Color Rules

Primary Chart Palette (Monochromatic)

For multi-series charts that don't carry semantic meaning (e.g., department comparison, monthly trends), use the blue tonal scale from 500 down to 100. This keeps charts monochromatic.

Semantic Chart Colors

For charts with status meaning (attendance, approvals, pipeline), use semantic colors. Always pair color with patterns or labels — never rely on color alone (colorblind accessibility).
